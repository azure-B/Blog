Data Structure - 자료구조

우리가 컴퓨터로 해결하려는 문제는 보통 데이터를 어떻게 다룰것인가? 로 귀결된다.
데이터를 무작위로 뿌려놓으면 필요 할 때 찾기도 어려울 뿐더러, 추가/삭제에 엄청난 시간이 걸린다.

올바른 자료구조의 선택은 시스템의 성능과 안정성을 좌우하는 핵심이며
데이터를 목적에 맞게, 가장 효율적으로 담고 꺼내 쓰기 위한 그릇이자 정리 정돈의 규칙이다

1. Stack - 스택

Stack은 논리적으로 LIFO ( Last in First Out )을 따르는 추상 자료형이다.
데이터의 메모리 삽입,삭제,읽기 등 접근이 한쪽 방향, TOP이라 부르는 단일 포인터에서만 진행된다.

추상 자료형 데이터의 구체적인 구현 방식은 숨기고, 자료구조의 기능(연산)과 특성만을 수학적으로 정의한 설계도

1-1. 스택의 내부 구현 방식과 메모리 구조

스택은 그 자체로 물리적인 메모리 구조라기 보다는 규칙을 정의한 인터페이스이다.
실제 메모리 상에선 배열이나 연결리스트를 기반으로 구현된다.

1-1-1. 배열 기반 스택

우선 연속적인 메모리 공간을 할당 받는다.
그리고 내부에서 정수형 Index인 TOP 변수를 유지하며 초기는 -1이다.
PUSH 연산시 TOP을 +1 하며, POP 연산시 Index Data를 반환하고 TOP을 -1한다.

캐시 지역성이 뛰어나 연속적인 접근이 빠르지만
지정된 배열의 크기에서 벗어나면
더 큰 배열을 선언하고 데이터를 복사하는 동적 배열 확장 비용이 발생한다.
이 경우에는 $O(n)$의 시간이 소요될 수 있다.

캐시 지역성 프로그램이 메모리에 접근할 때 시간적, 공간적으로 가까운 데이터를 집중적으로 참조하는 경향

1-2. 연결 리스트 기반 스택

메모리의 흩어진 공간에 동적으로 노드를 할당하고,
각 노드가 다음 노드의 포인터를 가지는 단방향 연결 리스트의 head 노드를 top으로 취급한다.

push와 pop은 단순히 포인터 연결만 변경하므로 항상 엄밀한 $O(1)$을 보장하며 크기 제한이 없다.
하지만 각 노드가 포인터를 위한 추가 메모리를 요구하고, 캐시 지역성이 떨어진다.

1-3. 구현예시

GitHub — Stack

Python의 List는 이미 동적 배열로 구현되어 있으며, 스택의 핵심 연산인 Push, Pop을
$O(1)$시간에 처리 할 수 있도록 되어있다.

객체 지향적 관점에서 스택의 동작만 하도록 클래스로 감싸보았다. 


      class Stack:
        def __init__(self):
            # 데이터를 담을 빈 리스트 초기화
            self._elements = []

        def push(self, item):
            """스택의 맨 위(끝)에 데이터를 추가합니다. O(1)"""
            self._elements.append(item)

        def pop(self):
            """스택의 맨 위(끝)에서 데이터를 꺼내고 반환합니다. O(1)"""
            if self.is_empty():
                raise IndexError("빈 스택에서 pop을 수행할 수 없습니다.")
            return self._elements.pop()

        def peek(self):
            """데이터를 꺼내지 않고 맨 위의 값을 확인만 합니다. O(1)"""
            if self.is_empty():
                raise IndexError("빈 스택에서 peek을 수행할 수 없습니다.")
            return self._elements[-1]

        def is_empty(self):
            """스택이 비어있는지 확인합니다."""
            return len(self._elements) == 0

        def __str__(self):
            return f"Stack(bottom -> top): {self._elements}"

      # 사용 예시
      my_stack = Stack()
      my_stack.push(10)
      my_stack.push(20)
      my_stack.push(30)
      print(my_stack)         # Stack(bottom -> top): [10, 20, 30]
      print(my_stack.pop())   # 30 반환

1-4. 알고리즘적 응용

1-4-1. 깊이 우선 탐색

DFS(깊이 우선 탐색)을 구현 할 때 보통 자기 자신을 호출하는 재귀 함수를 만든다.
하지만 파이썬과 같은 언어에선 재귀 호출이 깊어지게 되면 호출 스택 한계에 부딪혀 RecursionError가 발생한다.
이를 방지하고 메모리를 제어하기 위해 명시적인 스택 자료구조를 직접 구현하여 사용한다.

재귀 함수 함수 내부에서 자기 자신을 다시 호출하여 문제를 해결하는 프로그래밍 기법

동작의 핵심은 시작 노드를 스택에 넣고, 스택에서 노드 하나를 꺼낸 뒤
그 노드와 연결된 아직 방문하지 않은 이웃 노드들을 모조리 스택에 넣게 되면,
방금 넣은 이웃 노드 중 하나가 다음 번에 바로 꺼내져 끝없이 깊은 곳으로 먼저 파고들게 된다.

트랜스포머나 GAN같은 복잡한 딥러닝 모델을 설계 할 때,
PyTorch나 TensorFlow 내부에서는 수많은 연산들이 방향성 비순환 그래프를 이룬다.
이 후 역전파 학습시 DFS를 응용한 위상 정렬 알고리즘이 스택을 활용해 미분 연산 순서를 연결한다.

1-4-2. 단조 스택

단조 스택은 알고리즘 최적화의 꽃으로, 특정 조건을 강제함으로써, 불필요한 반복을 없앴다.
어떤 값이든 스택에 딱 한 번 Push,POP이 실행되어 배열을 뒤적거릴 필요 없이 계산이 끝난다.

2. Queue - 큐

큐는 데이터의 순서를 엄격하게 보존하는 FIFO구조의 자료 구조이다.
가장 먼저 들어온 데이터가 먼저 나간다는 원리지만,
이를 성능 손실 없이 구현하고 분산, 탐색에 적용하는 아주 정교한 기법들을 가지고있다.

2-1. 메모리 관리 매커니즘

큐는 데이터를 뺄 때 마다 모든 데이터를 한칸 앞으로 당겨와야 하는 연산 비용 문제 $O(n)$가 있다.
이 문제를 어떻게 극복했는지 그 원리를 이해하는 것이 중요하다.

2-1-1. 원형 큐 ( Circular Queue )

메모리 공간을 일직선이 아닌, 끝과 시작이 이어진 원으로 취급한다.
데이터를 꺼내서 앞부분에 빈 공간이 생기면,
뒤에서 자료를 밀어넣는 Rear 포인터가 배열의 끝에 도달 했을 때 멈추지 않고 순환한다.
데이터를 물리적으로 이동시키는 대신, 포인터의 시선만을 이동시켜 완벽한 효율 $O(1)$을 달성했다.

2-1-2. 연결 리스트 큐 ( Linked List Queue )

데이터를 연속된 공간에 넣는 것을 포기하고,
메모리 곳곳에 흩어진 데이터들이 서로 앞뒤의 주소만 서로 기억하게 한다.
맨 앞의 데이터를 꺼내면, 두번째 데이터를 포인터로 연결시켜 갱신만 하면 끝이다.
메모리 단편화는 존재하지만, 크기의 한계와 삽입/삭제의 지연을 완벽히 없앤 구조이다.

2-2. 큐의 알고리즘 응용

들어온 순서를 보존하는 큐는 상태공간 탐색등에 대체 불가능한 도구이다.

2-2-1. 너비 우선 탐색

스택이 DFS 방식에 유용하다면, 큐는 Root에서 시작점이 같은 노드를 확장하며 탐색한다.

큐에 현재 노드를 넣고, 노드를 꺼내면서 연결된 미방문 노드들을 줄지어 세워 탐색하는 구조이다.

가중치가 없는 그래프에서 최단 경로를 찾는 유일하고 완벽한 해법이다.

2-3. 시스템 아키텍쳐에서 큐

2-3-1. 메세지 큐

사용자의 요청 즉 Traffic이 쏟아질 때, 서버가 이를 모두 처리하려 들면 데이터베이스가 다운된다.
이 때 요청들을 Queue에 차곡차곡 담아 서버의 처리능력에 맞게 하나하나씩 꺼내어 처리한다.

2-3-2. AI Pipeline의 Data Prefetch

Data Prefetch
시스템이 필요할 것으로 예측되는 데이터를 미리 가져와 캐시에 저장하는 최적화 기법이다.

GAN이나 Diffusion과 같은 무거운 딥러닝 모델을 훈련 할 때,
GPU의 행렬 연산은 디스크에서 데이터를 읽어오는 IO속도보다 압도적으로 빠르다.
GPU가 연산을 마칠 때 마다 다음 데이터를 기다리게 하면, 치명적인 자원 낭비가 생기게 된다.
CPU는 다음 학습에 쓰일 미니 배치들을 미리 전처리하여 메모리 상의 큐에 끊임없이 적재해 둔다.
GPU는 연산이 끝나는 즉시 큐의 맨 앞에서 다음 데이터를 꺼내어 쉬지 않고 학습을 이어갈 수 있다.

코드 구현은 아래와 같다. 


      import tensorflow as tf

      # 1. 최대 10개의 정수(int32)를 담을 수 있는 선입선출(FIFO) 큐 생성
      fifo_queue = tf.queue.FIFOQueue(capacity=10, dtypes=tf.int32)

      # 2. 생산자 (Producer): 큐의 뒤(Rear)에 데이터를 밀어 넣습니다. (Enqueue)
      fifo_queue.enqueue(1)
      fifo_queue.enqueue(2)
      fifo_queue.enqueue_many([3, 4, 5]) # 여러 개를 한 번에 넣을 수도 있습니다.

      print(f"현재 큐에 쌓인 데이터 개수: {fifo_queue.size().numpy()}개")

      # 3. 소비자 (Consumer): 큐의 앞(Front)에서 데이터를 꺼냅니다. (Dequeue)
      # 가장 먼저 넣었던 '1'이 가장 먼저 나옵니다.
      print(f"첫 번째 Dequeue: {fifo_queue.dequeue().numpy()}")
      print(f"두 번째 Dequeue: {fifo_queue.dequeue().numpy()}")

2-4. 구현 예시

GitHub — Queue

Python list 이용하여 Queue 구현시 성능 문제가 발생하기에,
원형 큐 기법이나, 이중 연결 리스트 사용 


      from collections import deque
      import time

      N = 100000

      # 1. 성능이 매우 떨어지는 List 기반 큐 (사용 금지)
      list_queue = list(range(N))
      start = time.time()
      while list_queue:
          list_queue.pop(0)  # O(n) 연산이 n번 반복되어 총 O(n^2) 소요
      print(f"List pop(0) 소요 시간: {time.time() - start:.4f}초")

      # 2. O(1) 성능을 보장하는 Deque (표준 접근법)
      deque_queue = deque(range(N))
      start = time.time()
      while deque_queue:
          deque_queue.popleft()  # O(1) 연산이 n번 반복되어 총 O(n) 소요
      print(f"Deque popleft() 소요 시간: {time.time() - start:.4f}초")
